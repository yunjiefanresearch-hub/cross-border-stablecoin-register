# Cross-Border Stablecoin Register

[![DOI](https://zenodo.org/badge/DOI/10.5281/zenodo.20730358.svg)](https://doi.org/10.5281/zenodo.20730358)

An **open, versioned, machine-readable** register of how jurisdictions regulate stablecoins —
mapped clause by clause across **fifteen dimensions** and **two doctrinal spines** (the
permitted-activity / yield boundary and securities classification), with a queryable
**analysis layer** (the composition-problem apparatus), a **directed corridor layer**, and a
**computed layer** that *derives* cross-border feasibility from rules and flags where the
derivation disagrees with the papers. Built from primary sources, citable by DOI.

**Focus (twelve jurisdictions):** US · Hong Kong · EU · UK · Singapore · Mainland China · Brazil · Switzerland · United Arab Emirates · Taiwan · Japan · South Korea  ·  **Doctrinal anchor:** United States (CLARITY Act §404 / GENIUS Act)
**Status:** v0.10.1 · **Cadence:** quarterly diffs + event patches · **License:** CC-BY-4.0 (data) / Apache-2.0 (code) — see [`LICENSING.md`](LICENSING.md)

---

## What this is — and what it is not
This is **not** another broad, shallow, proprietary "200-country" tracker, and it is **not** a scrape.
It is a focused, **depth-first** reference whose value is the **judgment in the data** — sourcing and
ambiguity resolution made transparent (see [`METHODOLOGY.md`](METHODOLOGY.md)) — and whose
differentiator is the **corridor layer**: what actually clears, and what breaks, at each regulatory
boundary along a cross-border flow (see [the corridor file](hk-br-usd-stablecoin-settlement.yaml)).

Each record is one verified `(jurisdiction × instrument × dimension)` fact: structured, sourced to a
primary instrument with a pinpoint, dated, version-stamped, and machine-readable against a published
[schema](record.schema.json).

## Coverage
The complete coverage matrix, the per-jurisdiction depth, the two-axis evidence-tier breakdown, and the
citable count are generated by `build.py` into [`COVERAGE.md`](COVERAGE.md), which is the single source
of truth for those figures. This README does not restate them, so it cannot fall out of step with the
build.

The register spans **twelve focus jurisdictions** across **fifteen dimensions** plus **two doctrinal
spines** (the permitted-activity / yield boundary and securities classification), depth-first: each
verified cell is one `(jurisdiction × instrument × dimension)` fact, sourced to a primary instrument
with a pinpoint and machine-checked against a published schema. **v0.4.0** is the twelve-jurisdiction
expansion, adding Switzerland, the UAE, Taiwan, Japan, and South Korea as full focus jurisdictions;
Taiwan's AML-registration layer is in force while its draft Virtual Asset Service Act provisions are
carried as draft-flagged `status: proposed` records. For Switzerland, yield to holders is permitted but
structurally constrained (the bank guarantee must cover any interest), the register's clearest
permission-cluster anchor on the yield spine, opposite the EU/HK/US prohibition. The full grid, with the
planned schedule for the cells still open, is in [`COVERAGE.md`](COVERAGE.md).

## Use
```bash
pip install -r requirements.txt
python scripts/build_analysis.py   # (re)builds the analysis layer (compatibility matrix, etc.)
python scripts/build_corridors.py  # (re)builds the directed corridor edges + declares cross-layer divergences
python scripts/compose.py          # (re)builds the computed layer (compose() — computed-vs-authored diff)
python build.py                    # validates records + corridor schema + cross-layer integrity, compiles dataset.json
python build_site.py               # regenerates the static landing page from dataset.json

python run_invariants.py           # read-only structural assertions over the built register
python run_negative_tests.py       # proves the validation gates BITE (breaks each on a throwaway copy)
```
Runs on any locale: every file read/write passes `encoding="utf-8"` and the scripts reconfigure stdout to
UTF-8, so `python build.py` works on a non-UTF-8 console (e.g. Windows GBK) without `-X utf8`.
Continuous integration runs all of the above on every push and pull request
([`.github/workflows/build.yml`](.github/workflows/build.yml)), so an invalid record, a corridor that
contradicts the matrix, or a stale derived artifact cannot be merged.

- Machine-readable dataset: [`dataset.json`](dataset.json) — records, corridors, **and the `analysis` layer (incl. `analysis.computed`)**
- Schemas (the standards): [`record.schema.json`](record.schema.json) · [`corridor.schema.json`](corridor.schema.json) · [`analysis.schema.json`](analysis.schema.json) · vocabulary: [`taxonomy.md`](taxonomy.md)
- Add a record: copy [`_TEMPLATE.yaml`](_TEMPLATE.yaml); see [`CONTRIBUTING.md`](CONTRIBUTING.md)

**Explore it.** [`index.html`](index.html) is a self-contained static site with the coverage
**matrix**, a sortable multi-filter **table**, and the **pairwise compatibility matrix** (click any
cell for the category, interaction sets, and binding constraint of that jurisdiction pair). Deploys
as-is to GitHub Pages or Netlify (no build step; publish the repo root).

**Query it from an agent — one line, nothing to clone:**

```json
{ "mcpServers": { "cbsr": { "command": "uvx", "args": ["cbsr-mcp"] } } }
```

The [`cbsr-mcp`](https://pypi.org/project/cbsr-mcp/) package wraps the dataset in typed tools over
the Model Context Protocol: node-layer tools (`query`, `compare_dimension`, `jurisdiction_profile`,
`search`, `coverage`, `get_corridor`, …), the **lawyer-citable** tool (`citable_law`), analysis-layer
tools (`compatibility`, `interaction_sets`, `architectural_patterns`, `open_questions`), **computing
tools** (`compose_corridor`, `explain_feasibility`, `verification_report`), and **time-engine tools**
(`compose_corridor(as_of=…)`, `corridor_timeline`, `event_calendar`), **substrate tools** (`constraint_substrate`, `compose_via_substrate`), the **verification worklist** (`verification_worklist`), and the **stakeholder projection** (`stakeholder_database`, `profile_for`), and the **edge layer** (`edge_coverage`, `corridor_skeleton`). See
[`MCP_SERVER.md`](MCP_SERVER.md).

## Evidence model — `claim_class` × `evidence_tier`, and the citable subset
Every record carries **two orthogonal evidence axes**, so a reader can separate *what kind of claim
this is* from *how well it is sourced*:

- **`claim_class`** — the **kind** of claim. `tier1_legal` is a **proposition of law**: what a binding
  instrument (statute, regulation, official guidance) requires, permits, or prohibits — a fact a lawyer
  could cite to the instrument. `tier2_operational` is a **market/operational fact**: what is actually
  live, who is registered, what products launched, what banking rails exist — properly read as-of-dated.
- **`evidence_tier`** — the **provenance strength**. `resolution_text` = confirmed against the official
  text; `mixed` = core point confirmed, some operational detail pending; `firm_summary` =
  practitioner-corroborated, pending the official-text check.

These are independent. A confirmed product launch is well-sourced (`resolution_text`) but is *not* law
(`tier2_operational`); a draft statutory provision is a legal claim (`tier1_legal`) but not yet binding
(`status: proposed`). The distinction follows the Corridor Atlas §7 split between primary-source legal
constraints and market-reported operability, which the Atlas warns must not be read at the same
confidence.

**The lawyer-citable subset** is the intersection that a lawyer or supervisor can cite as current
binding law:

```
claim_class == tier1_legal  AND  status == in_force  AND  evidence_tier == resolution_text
```

Its current size is generated into [`COVERAGE.md`](COVERAGE.md); the v0.9.5 external verification pass promoted the in-force EU MiCA, US GENIUS, and HK Cap. 656 cells into it. It is precomputed as `citable_subset` in [`dataset.json`](dataset.json)
(each row projected to instrument + pinpoint + official URL), exposed by the MCP `citable_law()` tool,
and surfaced by the **"citable law only"** toggle in the static-site table. The build **enforces** it on
two axes: a `tier1_legal` record at `resolution_text`/`mixed` that asserts official-text confirmation
but lacks a `source.url` (or, at `resolution_text`, a `pinpoint`) fails the build; and a citable
record's `source.primary`/`pinpoint` may carry **only the legal instrument** — no product launch,
admission, or named commercial counterparty (`check_citable_purity`). So the subset cannot silently
drift away from "confirmed against the source," and the instrument a lawyer is shown never bundles a
market fact. The two-axis count matrix is in [`COVERAGE.md`](COVERAGE.md).

An operational illustration that contextualises a legal claim — a launch, a registration, a working
example — lives in the optional **`operational_notes`** field (Tier-2, as-of-dated), which the citable
projection omits. v0.7.1 moved exactly such material (the USDC / SBI VC Trade admission) out of the
load-bearing fields of `jp-epi-distribution-001`, which had been projecting it as citable law.

### A third axis — `binding_status` (what caps citability)

The v0.9.5 external verification pass added a third, orthogonal axis: **`binding_status`** — the binding
status of the *instrument* a cell rests on, distinct from how well-sourced it is (`evidence_tier`) and
whether the claim is legal or operational (`claim_class`). Its values: `in_force_enacted`,
`made_not_commenced`, `finalized_policy_pending`, `pending_proposal`, `prohibition`, `no_regime`. The
point is that **finding the official text is not enough to make a cell citable as current law** — the
instrument must actually be enacted and in force. So the build enforces: a `resolution_text` cell *must*
be `in_force_enacted`. This is why the UK SI 2026/102 cells stay below `resolution_text` even though the
text was read (the regime commences 2027-10-25), and why the SG MAS SCS-framework cells (finalized
policy, pending legislation) and the US CLARITY/NPRM cells are held at `firm_summary`. The pass is
recorded in [`analysis/verification_ledger.json`](analysis/verification_ledger.json) (exposed by the MCP
`verification_ledger()` tool), and a build gate cross-checks every record against it to catch drift.

The same discipline propagates to the computed layer: `compose()` derives feasibility only from
`tier1_legal` records (a signal that rested on a `tier2_operational` fact fails the build). This is the
foundation the **time/event engine** (v0.8.0) and the **constraint substrate** (v0.9.0) build on — a
date-aware or constraint-level verdict must distinguish legal facts from operational ones. The
time/event engine is sequenced **first**: the substrate's prerequisites (the evidence split and the
signal-provenance gate) are exactly what v0.6.0–v0.7.1 built, so the date engine is the now-unblocked,
highest-applicability, lowest-risk next step, while the substrate is the deeper, later thesis.

> **The quiet liability.** Every `compose()` result is only as honest as the hand-curated signal table
> and the cells beneath it, and a standing set of `tier1_legal` cells are still `unset` (transcribed,
> not yet confirmed against the official text). `claim_class` and the citable gates are the right
> *defenses*, but they do not retire that liability; **only the primary-source verification pass does.**
> The exact count, and the per-cell worklist, are tracked in the verification queue
> ([`COVERAGE.md`](COVERAGE.md), MCP `verification_report()`).

## Analysis layer — the composition-problem apparatus, as data
The Compliance Matrix supplies the **node** facts (one jurisdiction × one dimension); the
Architecture working paper supplies the **analysis** of how those nodes *compose* across borders.
As of **v0.5.0** that analysis is queryable data, not just prose, under `analysis` in
[`dataset.json`](dataset.json) and as standalone files in [`analysis/`](analysis):

- **`compatibility.json`** — the §5.14 **pairwise compatibility matrix**: every unordered
  pair of the twelve focus jurisdictions, each with its category (I dual-authorization / I/II hybrid / II partnership /
  III composition-problem-unresolved), the operative interaction sets, a Category-III axis where it
  applies (prohibition / pre-regime / counterparty-conditional), and the binding constraint.
- **`interaction_sets.json`** — the **six constraint-interaction sets** (§2.9, A–F): the constraint
  pair, the joint-binding mechanism, and a worked example for each.
- **`architectural_patterns.json`** — the **three-pattern PRC typology** (§3.3), the portable
  **three-layer routing architecture** (§4/§6) with the §4.4 five-factor operational test, and the
  six design principles.
- **`open_questions.json`** — the **open questions** (§7.1–§7.5) with conditional-status flags.

This is the register's answer to "the headline contribution lives only in the PDF": the
composition-problem analysis is now machine-queryable alongside the node facts.

## Corridor layer — directed edges
Cross-border treatment is **directional** (the inbound gate is the binding one), so corridors are
directed origin→destination edges. v0.5.0 expands the layer from the single worked example to a
set of hand-authored edges (US↔EU, US↔HK, SG→HK, HK→CN, US↔JP, plus the HK→BR anchor), each carrying the
Corridor Atlas attributes: feasibility class, compatibility category (linked to §5.14), live
archetypes (RC/SC/TC/DC), operative interaction sets, infrastructure overlap (Agora / mBridge /
Ensemble), the inbound mechanism and its administrator, and any origin-override flag. The asymmetry
is explicit — e.g. **US→EU is Category I but EU→US is Category II** (the §18 comparability gate),
and **HK→CN is blocked at destination**. Source: Corridor Atlas v0.2.3; the rest of the directed-edge space is
filled by the computed edge layer (depth before breadth).

## Computed layer — the composition problem as a running function
v0.6.0 took the first step past *storing* the analysis to *computing* it. The analysis layer and the
corridor layer were two independent transcriptions (the §5.14 matrix; the Atlas edges), linked by a
`compatibility_pair` foreign key but never reconciled — so most of the hand-authored corridors had interaction
sets that silently contradicted the matrix row they pointed to. Two mechanisms close that gap:

- **Cross-layer integrity (enforced in `build.py`, run in CI).** Every corridor carrying a
  `compatibility_pair` must have a category that **matches** its §5.14 row, and if its interaction
  sets differ they must carry an explicit **`divergence`** field explaining why (a directed edge read
  at the inbound gate legitimately differs from an undirected pair — but the difference is now
  *declared*, not silent). The build fails otherwise. The seven contradictions are now seven declared
  divergences; `hk-us` and `hk-br` match exactly.
- **`compose()` (`scripts/compose.py` → `analysis/computed_compatibility.json`).** A small, auditable
  rule engine *derives* each pair's feasibility from a per-jurisdiction signal table (regime status,
  inbound-gate type, exportable-token, egress override — each annotated with the node record that
  justifies it) using the Atlas's own algorithm (origin drag, then destination-determined class), and
  **diffs computed-vs-authored**. It reproduces every authored corridor and all but a handful of the §5.14
  categories from first principles; the `pre_regime` set it derives from the signals matches the set
  read directly off the records. **The residual disagreements are a finding, not a bug**: both are the
  `uk_regime_in_transition` cause — the Atlas flags the UK as not operative until 2027 while the
  Architecture's §5.14 treats US-UK and EU-UK as cleanly bridgeable Category I. That divergence is
  exactly the research-valuable signal the engine exists to surface. The layer is a labelled
  **preview**, not asserted authoritative; it ships under `analysis.computed` in `dataset.json` and is
  exposed by the MCP tools `compose_corridor` and `explain_feasibility`. Corridors validate against a
  real **`corridor.schema.json`** (two shapes: the flat directed edge and the rich `hk-br` analysis),
  enforced like the node schema.

## Time engine — feasibility over time (v0.8.0)
Corridor feasibility is **dated**. `compose(origin, destination, as_of)` applies every scheduled change
in law effective by a date and re-runs the Atlas algorithm, so the same edge can read one class today
and another once a regime is operative.

- **Event calendar** ([`analysis/event_calendar.json`](analysis/event_calendar.json)): dated and
  contingent **changes in law** that move a jurisdiction's signal — `scheduled` (an `effective_date`,
  optionally `precision: year`), `contingent` (a bill with no firm date — never applied by date, only
  surfaced as an "if enacted" branch), and `in_force`. **Every event is `claim_class: tier1_legal`,
  backed by `tier1_legal` records — enforced by the build.** A market launch is never an event; that is
  the v0.7 split inherited into the time dimension.
- **The headline (honest version):** the directed edge **US→UK is Category T today and Category I as of
  2027**, when the UK systemic regime is operative; the regime-in-transition caveat on UK pairs resolves. But the `EU-UK` and `UK-US` undirected pairs **remain findings after 2027**
  — their residual is *structural* (the Atlas scores live-live as `I/II`; §5.14 authored them as
  cleanly bridgeable `I`), not temporal. The engine separates a regime-in-transition artifact that
  resolves by date from a genuine modelling difference that does not.
- Ships under `analysis.event_calendar` and `analysis.computed_timeline` in `dataset.json`; exposed by
  the MCP tools `compose_corridor(as_of=…)`, `corridor_timeline`, and `event_calendar`. This is feature
  #1 of the three-feature plan; the constraint substrate (#3) is next (v0.9.0).

## Constraint substrate — feasibility composed from C1–C8 (v0.9.0)
The deepest layer. Each jurisdiction's stance on the eight constraints is a structured **pole**, and
`compose_via_substrate(origin, destination)` derives a corridor's class by composing two jurisdictions'
poles **through the six interaction-set rules** (§2.9) — origin drag, the C7×C1 destination gate, and
the Set A data-sovereignty overlay — not by reading a single inbound gate.

- **Provenance, one level deeper:** every pole cites the `tier1_legal` record it is transcribed from;
  the build rejects a pole backed by a market record. A pole exists only where a record backs it.
- **It never guesses:** where a load-bearing pole is unset, the engine returns `indeterminate` with the
  missing poles, so coverage reports the backlog honestly. With the constraint cells
  authored across all twelve jurisdictions, the substrate derives every authored corridor and almost all
  directed edges, returning `indeterminate` only for the edges into the UK while it is in transition; the
  exact cell and edge coverage is generated into [`COVERAGE.md`](COVERAGE.md).
- **Cross-checked:** where the substrate yields a definite class it must equal the signal-table
  `compose()`, enforced across all directed edges. Worked derivations: **HK→JP = II** (Japan's
  *channelled* C7), **JP→HK = I** (Hong Kong's *open* C7), **US→EU = I**, **EU→US = II** — all agreeing.
- Ships under `analysis.constraint_substrate` / `analysis.computed_substrate`; exposed by
  `constraint_substrate` and `compose_via_substrate`. Populating the remaining cells and the
  primary-source verification pass — which lights the substrate up and retires the standing liability —
  is the v0.9.x backlog.

## Edge layer — derived skeletons for the rest of the corridor space

The hand-authored corridor records are the enriched gold tier (they carry `infrastructure_overlap`,
bespoke inbound detail, curated archetypes, prose). For every *other* derivable directed edge, the build
emits a **computed skeleton** — the structurally-derivable fields only: feasibility class (from the
substrate), inbound-mechanism `test` + `administrator` (re-projected from the destination's C1/C7 record),
a baseline archetype set, the directed interaction sets, and provenance. With these skeletons the edge
layer now has a record for almost all directed edges (the ones still without are into the UK, in
transition); the exact skeleton and edge count is generated into [`COVERAGE.md`](COVERAGE.md). No new
facts: every skeleton's class is cross-checked against the signal `compose()` and cites its backing
`tier1_legal` records, and the empirical fields (`infrastructure_overlap`, bespoke detail) are left unset
as a per-edge enrichment backlog. Ships under `analysis.computed_corridor_skeletons`, exposed by the MCP
tools `edge_coverage` and `corridor_skeleton`.

## Stakeholder projection (Atlas §8) — the same corridor, seen by each actor

Each persona reads a different slice of C1–C8. `profile_for(stakeholder, edge)` re-projects the
*already-derived* corridor class and the substrate poles through that lens — issuer, distributor/wallet,
reserve custodian, home/host regulator, corporate treasury, institutional holder. It introduces **no new
legal facts**: every projected line is read from an existing record and cites it, the build fails if a
profile rests on a missing or non-`tier1_legal` record, and a profile stays **preview**, bounded by the
verification status of the cells it reads. Ships under `analysis.stakeholder_database` and
`analysis.computed_stakeholder_profiles` (one worked profile per authored corridor and persona), exposed
by the MCP tools `stakeholder_database` and `profile_for`.

## Verification — earned tiers, and the scoped pass (v0.9.1)
The substrate's coverage is bounded by a backlog stated since v0.7: a standing set of `tier1_legal`
cells are unverified (the exact count is generated into [`COVERAGE.md`](COVERAGE.md)), and every
`compose()`/substrate result rests on them. v0.9.1 ships the harness that makes
the verification pass safe and scoped — not the (external) verification itself, which is never
fabricated:

- **`evidence_tier` is earned, not asserted.** The build enforces necessary conditions per tier
  (`resolution_text` ⟹ `url` + `pinpoint` + `last_reviewed`; `mixed` ⟹ `url` + `pinpoint`;
  `firm_summary` ⟹ `pinpoint`). A cell can no longer claim a tier it lacks the provenance for.
- **A per-cell worklist** (`analysis.verification_worklist`, tool `verification_worklist`): for each
  unverified cell, the instrument + pinpoint and exactly what is missing, every one currently lacking a
  `source.url`, concentrated in HK/SG/UK/CN/US/EU.
- **An audit trail.** The optional `verification` block records *how* a cell was confirmed
  (`method`, `against`, `verified_on`), enabling a safe, traceable promotion.

Confirming these cells against the official text promotes them to `resolution_text` and confirms the authored pole assignments, and retires the
liability; it requires primary sources and is deliberately not faked here.

## Name & identifiers
The canonical name is **`Cross-Border Stablecoin Register` (CBSR)** — it foregrounds the
differentiator (a cross-jurisdictional comparison, not a single-jurisdiction tracker) and reads as a
standing reference object rather than a payments product. The citable anchor is the **DOI**, not a
domain.

The repository slug is **`cross-border-stablecoin-register`**, and every live identifier now follows
it: the three schema `$id`s, `CITATION.cff`, `mcp.json`, the MCP client path examples, and the
generated landing page. The project was originally `stablecoin-rail-register`; through v0.10.1 the
schema `$id`s still pointed at that slug's GitHub Pages path, which the rename had already made
non-canonical — a stale identifier of exactly the kind this register exists to catch. The published
schemas are now self-consistent with the repository that publishes them, and
`tools/check_identifiers.py` fails the build if they ever diverge again.

Two places still say `stablecoin-rail-register`, deliberately: `CHANGELOG.md` and the v0.5.0 build
note. Those are dated records of what was true when they were written, and rewriting history to
match the present is precisely the failure mode the evidence discipline forbids.
