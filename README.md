# Cross-Border Stablecoin Register

[![DOI](https://zenodo.org/badge/DOI/10.5281/zenodo.20730358.svg)](https://doi.org/10.5281/zenodo.20730358)

An open, versioned, machine-readable register of how jurisdictions regulate stablecoins. It maps regulation clause by clause across fifteen dimensions and two doctrinal spines (the permitted-activity / yield boundary and securities classification). Alongside the records it provides a queryable analysis layer for the composition problem, a directed corridor layer, and a computed layer that derives cross-border feasibility from the rules and flags where that derivation disagrees with the companion papers. It is built from primary sources and citable by DOI.

**Focus (twelve jurisdictions):** US · Hong Kong · EU · UK · Singapore · Mainland China · Brazil · Switzerland · United Arab Emirates · Taiwan · Japan · South Korea  ·  **Doctrinal anchor:** United States (CLARITY Act §404 / GENIUS Act)
**Status:** v0.10.1 · **Cadence:** quarterly diffs + event patches · **License:** CC-BY-4.0 (data) / Apache-2.0 (code)

## What this is, and what it is not

This is a focused, depth-first reference rather than a broad "200-country" tracker or a scrape. Its value lies in the judgment in the data, with sourcing and ambiguity resolution made transparent (see [`METHODOLOGY.md`](METHODOLOGY.md)). Its main differentiator is the corridor layer, which records what actually clears and what breaks at each regulatory boundary along a cross-border flow (see [the corridor file](hk-br-usd-stablecoin-settlement.yaml)).

Each record is one verified `(jurisdiction × instrument × dimension)` fact. Every record is structured, sourced to a primary instrument with a pinpoint, dated, version-stamped, and machine-readable against a published [schema](record.schema.json).

This README does not restate them, so it cannot fall out of step with the build.

The register spans twelve focus jurisdictions across fifteen dimensions plus two doctrinal spines (the permitted-activity / yield boundary and securities classification), depth-first. Each verified cell is one `(jurisdiction × instrument × dimension)` fact, sourced to a primary instrument with a pinpoint and machine-checked against a published schema. v0.4.0 is the twelve-jurisdiction expansion, adding Switzerland, the UAE, Taiwan, Japan, and South Korea as full focus jurisdictions. Taiwan's AML-registration layer is in force while its draft Virtual Asset Service Act provisions are carried as draft-flagged `status: proposed` records. For Switzerland, yield to holders is permitted but structurally constrained (the bank guarantee must cover any interest), making it the register's clearest permission-cluster anchor on the yield spine, opposite the EU/HK/US prohibition. The full grid, with the planned schedule for the cells still open, is in [`COVERAGE.md`](COVERAGE.md).

## Use

```bash
pip install -r requirements.txt
python scripts/build_analysis.py   # (re)builds the analysis layer (compatibility matrix, etc.)
python scripts/build_corridors.py  # (re)builds the directed corridor edges + declares cross-layer divergences
python scripts/compose.py          # (re)builds the computed layer (compose(), the computed-vs-authored diff)
python build.py                    # validates records + corridor schema + cross-layer integrity, compiles dataset.json
python build_site.py               # regenerates the static landing page from dataset.json

python run_invariants.py           # read-only structural assertions over the built register
python run_negative_tests.py       # proves the validation gates BITE (breaks each on a throwaway copy)
```

Runs on any locale: every file read and write passes `encoding="utf-8"` and the scripts reconfigure stdout to UTF-8, so `python build.py` works on a non-UTF-8 console (e.g. Windows GBK) without `-X utf8`. Continuous integration runs all of the above on every push and pull request ([`.github/workflows/build.yml`](.github/workflows/build.yml)), so an invalid record, a corridor that contradicts the matrix, or a stale derived artifact cannot be merged.

- Machine-readable dataset: [`dataset.json`](dataset.json), containing records, corridors, and the `analysis` layer (including `analysis.computed`)
- Schemas (the standards): [`record.schema.json`](record.schema.json) · [`corridor.schema.json`](corridor.schema.json) · [`analysis.schema.json`](analysis.schema.json) · vocabulary: [`taxonomy.md`](taxonomy.md)
- Add a record: copy [`_TEMPLATE.yaml`](_TEMPLATE.yaml); see [`CONTRIBUTING.md`](CONTRIBUTING.md)

**Explore it.** [`index.html`](index.html) is a self-contained static site with the coverage matrix, a sortable multi-filter table, and the pairwise compatibility matrix (click any cell for the category, interaction sets, and binding constraint of that jurisdiction pair). It deploys as-is to GitHub Pages or Netlify with no build step; publish the repo root.

**Query it from an agent.** [`mcp_server.py`](mcp_server.py) wraps the dataset in typed tools over the Model Context Protocol: node-layer tools (`query`, `compare_dimension`, `jurisdiction_profile`, `search`, `coverage`, `get_corridor`, and others), the lawyer-citable tool (`citable_law`), analysis-layer tools (`compatibility`, `interaction_sets`, `architectural_patterns`, `open_questions`), computing tools (`compose_corridor`, `explain_feasibility`, `verification_report`), time-engine tools (`compose_corridor(as_of=…)`, `corridor_timeline`, `event_calendar`), substrate tools (`constraint_substrate`, `compose_via_substrate`), the verification worklist (`verification_worklist`), the stakeholder projection (`stakeholder_database`, `profile_for`), and the edge layer (`edge_coverage`, `corridor_skeleton`). See [`MCP_SERVER.md`](MCP_SERVER.md).

## Evidence model: `claim_class` × `evidence_tier`, and the citable subset

Every record carries two orthogonal evidence axes, so a reader can separate what kind of claim it is from how well it is sourced.

- `claim_class` is the kind of claim. `tier1_legal` is a proposition of law: what a binding instrument (statute, regulation, official guidance) requires, permits, or prohibits, a fact a lawyer could cite to the instrument. `tier2_operational` is a market or operational fact: what is actually live, who is registered, what products launched, what banking rails exist, properly read as-of-dated.
- `evidence_tier` is the provenance strength. `resolution_text` means confirmed against the official text; `mixed` means the core point is confirmed with some operational detail pending; `firm_summary` means practitioner-corroborated, pending the official-text check.

These axes are independent. A confirmed product launch is well-sourced (`resolution_text`) but is not law (`tier2_operational`); a draft statutory provision is a legal claim (`tier1_legal`) but not yet binding (`status: proposed`). The distinction follows the Corridor Atlas §7 split between primary-source legal constraints and market-reported operability, which the Atlas warns must not be read at the same confidence.

The lawyer-citable subset is the intersection that a lawyer or supervisor can cite as current binding law:

```
claim_class == tier1_legal  AND  status == in_force  AND  evidence_tier == resolution_text
```

Its current size is generated into [`COVERAGE.md`](COVERAGE.md); the v0.9.5 external verification pass promoted the in-force EU MiCA, US GENIUS, and HK Cap. 656 cells into it. It is precomputed as `citable_subset` in [`dataset.json`](dataset.json) (each row projected to instrument plus pinpoint plus official URL), exposed by the MCP `citable_law()` tool, and surfaced by the "citable law only" toggle in the static-site table. The build enforces it on two axes. First, a `tier1_legal` record at `resolution_text` or `mixed` that asserts official-text confirmation but lacks a `source.url` (or, at `resolution_text`, a `pinpoint`) fails the build. Second, a citable record's `source.primary` and `pinpoint` may carry only the legal instrument, with no product launch, admission, or named commercial counterparty (`check_citable_purity`). So the subset cannot silently drift away from "confirmed against the source," and the instrument a lawyer is shown never bundles a market fact. The two-axis count matrix is in [`COVERAGE.md`](COVERAGE.md).

An operational illustration that contextualises a legal claim, such as a launch, a registration, or a working example, lives in the optional `operational_notes` field (Tier-2, as-of-dated), which the citable projection omits. v0.7.1 moved exactly such material (the USDC / SBI VC Trade admission) out of the load-bearing fields of `jp-epi-distribution-001`, which had been projecting it as citable law.

### A third axis: `binding_status`, which caps citability

The v0.9.5 external verification pass added a third, orthogonal axis, `binding_status`, the binding status of the instrument a cell rests on, distinct from how well-sourced it is (`evidence_tier`) and whether the claim is legal or operational (`claim_class`). Its values are `in_force_enacted`, `made_not_commenced`, `finalized_policy_pending`, `pending_proposal`, `prohibition`, and `no_regime`. The point is that finding the official text is not enough to make a cell citable as current law; the instrument must actually be enacted and in force. So the build enforces a rule: a `resolution_text` cell must be `in_force_enacted`. This is why the UK SI 2026/102 cells stay below `resolution_text` even though the text was read (the regime commences 2027-10-25), and why the SG MAS SCS-framework cells (finalized policy, pending legislation) and the US CLARITY/NPRM cells are held at `firm_summary`. The pass is recorded in [`analysis/verification_ledger.json`](analysis/verification_ledger.json) (exposed by the MCP `verification_ledger()` tool), and a build gate cross-checks every record against it to catch drift.

The same discipline propagates to the computed layer: `compose()` derives feasibility only from `tier1_legal` records, and a signal that rested on a `tier2_operational` fact fails the build. This is the foundation the time/event engine (v0.8.0) and the constraint substrate (v0.9.0) build on, since a date-aware or constraint-level verdict must distinguish legal facts from operational ones. The time/event engine is sequenced first: the substrate's prerequisites (the evidence split and the signal-provenance gate) are exactly what v0.6.0 through v0.7.1 built, so the date engine is the now-unblocked, highest-applicability, lowest-risk next step, while the substrate is the deeper, later thesis.

> **The standing liability.** Every `compose()` result is only as honest as the hand-curated signal table and the cells beneath it, and a standing set of `tier1_legal` cells are still `unset` (transcribed, not yet confirmed against the official text). `claim_class` and the citable gates are the right defenses, but they do not retire that liability; only the primary-source verification pass does. The exact count, and the per-cell worklist, are tracked in the verification queue ([`COVERAGE.md`](COVERAGE.md), MCP `verification_report()`).

## Analysis layer: the composition-problem apparatus as data

The Compliance Matrix supplies the node facts (one jurisdiction × one dimension); the Architecture working paper supplies the analysis of how those nodes compose across borders. As of v0.5.0 that analysis is queryable data rather than only prose, under `analysis` in [`dataset.json`](dataset.json) and as standalone files in [`analysis/`](analysis):

- `compatibility.json` is the §5.14 pairwise compatibility matrix: every unordered pair of the twelve focus jurisdictions, each with its category (I dual-authorization, I/II hybrid, II partnership, III composition-problem-unresolved), the operative interaction sets, a Category-III axis where it applies (prohibition, pre-regime, counterparty-conditional), and the binding constraint.
- `interaction_sets.json` holds the six constraint-interaction sets (§2.9, A-F): the constraint pair, the joint-binding mechanism, and a worked example for each.
- `architectural_patterns.json` holds the three-pattern PRC typology (§3.3), the portable three-layer routing architecture (§4/§6) with the §4.4 five-factor operational test, and the six design principles.
- `open_questions.json` holds the open questions (§7.1-§7.5) with conditional-status flags.

With this, the composition-problem analysis is machine-queryable alongside the node facts, rather than living only in the PDF.

## Corridor layer: directed edges

Cross-border treatment is directional (the inbound gate is the binding one), so corridors are directed origin→destination edges. v0.5.0 expands the layer from the single worked example to a set of hand-authored edges (US↔EU, US↔HK, SG→HK, HK→CN, US↔JP, plus the HK→BR anchor), each carrying the Corridor Atlas attributes: feasibility class, compatibility category (linked to §5.14), live archetypes (RC/SC/TC/DC), operative interaction sets, infrastructure overlap (Agora / mBridge / Ensemble), the inbound mechanism and its administrator, and any origin-override flag. The asymmetry is explicit. For example, US→EU is Category I but EU→US is Category II (the §18 comparability gate), and HK→CN is blocked at destination. Source: Corridor Atlas v0.2.3; the rest of the directed-edge space is filled by the computed edge layer, prioritising depth over breadth.

## Computed layer: the composition problem as a running function

v0.6.0 took the first step past storing the analysis to computing it. The analysis layer and the corridor layer were two independent transcriptions (the §5.14 matrix and the Atlas edges), linked by a `compatibility_pair` foreign key but never reconciled, so most of the hand-authored corridors had interaction sets that silently contradicted the matrix row they pointed to. Two mechanisms close that gap:

- Cross-layer integrity (enforced in `build.py`, run in CI). Every corridor carrying a `compatibility_pair` must have a category that matches its §5.14 row, and if its interaction sets differ they must carry an explicit `divergence` field explaining why. (A directed edge read at the inbound gate legitimately differs from an undirected pair, but the difference is now declared rather than silent.) The build fails otherwise. The seven contradictions are now seven declared divergences; `hk-us` and `hk-br` match exactly.
- `compose()` (`scripts/compose.py` → `analysis/computed_compatibility.json`) is a small, auditable rule engine that derives each pair's feasibility from a per-jurisdiction signal table (regime status, inbound-gate type, exportable-token, egress override, each annotated with the node record that justifies it) using the Atlas's own algorithm (origin drag, then destination-determined class), and diffs computed against authored. It reproduces every authored corridor and all but a handful of the §5.14 categories from first principles; the `pre_regime` set it derives from the signals matches the set read directly off the records. The residual disagreements are a finding rather than a bug: both are the `uk_regime_in_transition` cause, where the Atlas flags the UK as not operative until 2027 while the Architecture's §5.14 treats US-UK and EU-UK as cleanly bridgeable Category I. That divergence is exactly the research-valuable signal the engine exists to surface. The layer is a labelled preview, not asserted as authoritative; it ships under `analysis.computed` in `dataset.json` and is exposed by the MCP tools `compose_corridor` and `explain_feasibility`. Corridors validate against a real `corridor.schema.json` (two shapes: the flat directed edge and the rich `hk-br` analysis), enforced like the node schema.

## Time engine: feasibility over time (v0.8.0)

Corridor feasibility is dated. `compose(origin, destination, as_of)` applies every scheduled change in law effective by a date and re-runs the Atlas algorithm, so the same edge can read one class today and another once a regime is operative.

- Event calendar ([`analysis/event_calendar.json`](analysis/event_calendar.json)): dated and contingent changes in law that move a jurisdiction's signal. These are `scheduled` (an `effective_date`, optionally `precision: year`), `contingent` (a bill with no firm date, never applied by date, only surfaced as an "if enacted" branch), or `in_force`. Every event is `claim_class: tier1_legal`, backed by `tier1_legal` records, and this is enforced by the build. A market launch is never an event; that is the v0.7 split inherited into the time dimension.
- The key result: the directed edge US→UK is Category T today and Category I as of 2027, when the UK systemic regime is operative and the regime-in-transition caveat on UK pairs resolves. But the `EU-UK` and `UK-US` undirected pairs remain findings after 2027, because their residual is structural (the Atlas scores live-live as `I/II`; §5.14 authored them as cleanly bridgeable `I`) rather than temporal. The engine separates a regime-in-transition artifact that resolves by date from a genuine modelling difference that does not.
- Ships under `analysis.event_calendar` and `analysis.computed_timeline` in `dataset.json`; exposed by the MCP tools `compose_corridor(as_of=…)`, `corridor_timeline`, and `event_calendar`. This is feature #1 of the three-feature plan; the constraint substrate (#3) is next (v0.9.0).

## Constraint substrate: feasibility composed from C1-C8 (v0.9.0)

This is the deepest layer. Each jurisdiction's stance on the eight constraints is a structured pole, and `compose_via_substrate(origin, destination)` derives a corridor's class by composing two jurisdictions' poles through the six interaction-set rules (§2.9): origin drag, the C7×C1 destination gate, and the Set A data-sovereignty overlay, rather than by reading a single inbound gate.

- Provenance, one level deeper: every pole cites the `tier1_legal` record it is transcribed from, and the build rejects a pole backed by a market record. A pole exists only where a record backs it.
- It returns `indeterminate` rather than guessing: where a load-bearing pole is unset, the engine returns `indeterminate` with the missing poles, so coverage reports the backlog honestly. With the constraint cells authored across all twelve jurisdictions, the substrate derives every authored corridor and almost all directed edges, returning `indeterminate` only for the edges into the UK while it is in transition. The exact cell and edge coverage is generated into [`COVERAGE.md`](COVERAGE.md).
- Cross-checked: where the substrate yields a definite class it must equal the signal-table `compose()`, enforced across all directed edges. Worked derivations: HK→JP = II (Japan's channelled C7), JP→HK = I (Hong Kong's open C7), US→EU = I, EU→US = II, all agreeing.
- Ships under `analysis.constraint_substrate` and `analysis.computed_substrate`; exposed by `constraint_substrate` and `compose_via_substrate`. Populating the remaining cells and the primary-source verification pass, which lights the substrate up and retires the standing liability, is the v0.9.x backlog.

## Edge layer: derived skeletons for the rest of the corridor space

The hand-authored corridor records are the enriched gold tier; they carry `infrastructure_overlap`, bespoke inbound detail, curated archetypes, and prose. For every other derivable directed edge, the build emits a computed skeleton: the structurally-derivable fields only, namely feasibility class (from the substrate), inbound-mechanism `test` and `administrator` (re-projected from the destination's C1/C7 record), a baseline archetype set, the directed interaction sets, and provenance. With these skeletons the edge layer now has a record for almost all directed edges (the ones still without are into the UK, in transition); the exact skeleton and edge count is generated into [`COVERAGE.md`](COVERAGE.md). No new facts are introduced: every skeleton's class is cross-checked against the signal `compose()` and cites its backing `tier1_legal` records, and the empirical fields (`infrastructure_overlap`, bespoke detail) are left unset as a per-edge enrichment backlog. Ships under `analysis.computed_corridor_skeletons`, exposed by the MCP tools `edge_coverage` and `corridor_skeleton`.

## Stakeholder projection (Atlas §8): the same corridor, seen by each actor

Each persona reads a different slice of C1-C8. `profile_for(stakeholder, edge)` re-projects the already-derived corridor class and the substrate poles through that lens: issuer, distributor/wallet, reserve custodian, home/host regulator, corporate treasury, institutional holder. It introduces no new legal facts: every projected line is read from an existing record and cites it, the build fails if a profile rests on a missing or non-`tier1_legal` record, and a profile stays preview, bounded by the verification status of the cells it reads. Ships under `analysis.stakeholder_database` and `analysis.computed_stakeholder_profiles` (one worked profile per authored corridor and persona), exposed by the MCP tools `stakeholder_database` and `profile_for`.

## Verification: earned tiers, and the scoped pass (v0.9.1)

The substrate's coverage is bounded by a backlog stated since v0.7: a standing set of `tier1_legal` cells are unverified (the exact count is generated into [`COVERAGE.md`](COVERAGE.md)), and every `compose()` or substrate result rests on them. v0.9.1 ships the harness that makes the verification pass safe and scoped, not the external verification itself, which is never fabricated.

- `evidence_tier` is earned, not asserted. The build enforces necessary conditions per tier (`resolution_text` requires `url` plus `pinpoint` plus `last_reviewed`; `mixed` requires `url` plus `pinpoint`; `firm_summary` requires `pinpoint`). A cell can no longer claim a tier it lacks the provenance for.
- A per-cell worklist (`analysis.verification_worklist`, tool `verification_worklist`) gives, for each unverified cell, the instrument plus pinpoint and exactly what is missing, every one currently lacking a `source.url`, concentrated in HK/SG/UK/CN/US/EU.
- An audit trail. The optional `verification` block records how a cell was confirmed (`method`, `against`, `verified_on`), enabling a safe, traceable promotion.

Confirming these cells against the official text promotes them to `resolution_text`, confirms the authored pole assignments, and retires the liability. It requires primary sources and is deliberately not faked here.

## Name and identifiers

The canonical name is `Cross-Border Stablecoin Register` (CBSR). It foregrounds the differentiator, a cross-jurisdictional comparison rather than a single-jurisdiction tracker, and reads as a standing reference object rather than a payments product. The citable anchor is the DOI, not a domain. One identifier intentionally lags the name: the repository slug and GitHub Pages URL remain `stablecoin-rail-register` (the project's original name), and the schema `$id` and the MCP path example follow that slug. The slug is retained deliberately so that existing repository links, the Zenodo DOI, and the published schema `$id` stay stable; renaming the repo would break inbound references and the schema identifier for no functional gain. Everywhere the project is named to a reader (this README, the `CITATION.cff` title, the `LICENSE` headers, the dataset `name`, the landing page) it is CBSR; where a URL or stable identifier is involved, the legacy slug is canonical.
