# Build note — v0.10.1: forward/sensitivity regeneration from the corrected corridor layer

**Type:** patch (consistency fix; no new legal facts).
**Date:** 2026-07-09.

## What this release fixes

v0.10.0 shipped an **internal inconsistency in the forward-looking layer**. The authoritative
directed-corridor layer (`analysis/computed_corridors_directed.json`) already scored the eight
inbound-US edges as **regime-in-transition (T)**, resolving to Category II when the GENIUS Act
**§18 foreign-issuer comparability gate commences** (dated outer cap **2027-01-18**, per each
edge's `as_of_timeline`). But three derived files still carried the *pre-commencement* framing —
they treated the US as if the §18 gate were already operative, left the US out of the sensitivity
ordering, and gave it no class-moving pending event:

| file | v0.10.0 (stale) | v0.10.1 (regenerated) |
|---|---|---|
| `computed_sensitivity.json` | ordering `KR 20 / UK 8 / TW 9`; US in the **insensitive** set (via `us-clarity-act-enacted`, 0 class edges) | ordering `KR 21 / TW 18 / UK 8 / US 8`; insensitive `SG, CN` |
| `computed_forward_view.json` | US `own_pending_events: []`; "US has no class-moving pending event of its own" | US carries `us-genius-act-effective` (§18 commencement, 2027-01-18); `own_driven_inbound: 8` |
| `computed_timeline.json` (edge_timelines) | inbound-US edges `today_class: II` | `today_class: T`; scheduled `us-genius-act-effective → II` |

The regeneration also updates the static API projections (`api/sensitivity.json`, `api/forward.json`)
and bumps `api/meta.json`.

## Note on GENIUS "in force"

The GENIUS Act's **issuer layer is in force** — that statement, which appears in the
`us-clarity-act-enacted` event's `basis` note, is correct and is *retained*. The narrow correction is
that the **§18 foreign-issuer comparability gate** (which governs whether a foreign token may be
admitted into the US) is not yet operative: GENIUS §20 sets its effective date at the earlier of 18
months after enactment (2027-01-18) or 120 days after final regulations, and at the snapshot only
proposed rules existed, so the 2027-01-18 outer cap binds. Inbound-US edges are therefore `T` today,
not `II`. CLARITY remains a *separate* contingent market-structure event that does not move corridor
class (`would_change: false`).

## Provenance / how it was regenerated

Regenerated by `scripts/regen_forward_sensitivity.py` from the register's own data — specifically the
frozen forward transition set at `analysis/forward_transition_set.json` (a first-class, version-controlled
register input) together with the directed-corridor layer. The complete forward transition set (scheduled
**and** contingent) is proven edge-for-edge identical to this register's corridor class layer (132/132),
and the step re-derives the per-jurisdiction sensitivity edge counts and the §4.4 forward view. Two
independent cross-checks gate the write (the step exits non-zero on either failure, so it is a build GATE):

1. every **scheduled** transition in the output matches this register's own per-edge `as_of_timeline`
   (`ALL MATCH`); and
2. the recomputed sensitivity edge counts equal the ordering written (`KR 21 / TW 18 / UK 8 / US 8`).

This step is wired into the canonical build pipeline (`.github/workflows/build.yml`), immediately after
`scripts/build_forward_view.py` and before `build_api.py`, so a full rebuild **reproduces** the corrected
views rather than reverting them; CI's `git diff --exit-code` gate therefore holds. The build is
self-contained: the correction no longer depends on the mapper frontend. (The previous
`tools/regen_forward_sensitivity.py`, which sourced the transition set from the mapper JSX, is retained
only as a superseded shim that delegates here and ignores the legacy JSX argument.)

`compute_reconciliation_report.json` (shipped alongside) shows the corridor class layer was already
reproducible (0 edge divergences); this release brings the sensitivity/forward/timeline views into
line with it.

## Honest residual (not yet regenerated)

- `computed_corridor_states.json` (and its `api/corridor_states.json` projection) use a **separate,
  date-keyed jurisdiction-state engine** rather than the per-directed-edge class layer. Its `date_states`
  have not been re-run against the §18-commencement framing, and `scripts/build_sensitivity.py` /
  `scripts/build_forward_view.py` — which read that engine — still emit the pre-commencement form; the
  pipeline's `scripts/regen_forward_sensitivity.py` step then supersedes their forward/sensitivity output
  with the corrected, cross-checked values from the frozen transition set. It is a secondary
  representation; the authoritative directed-edge layer and the regenerated sensitivity/forward views are
  the load-bearing ones and are now correct. Folding the §18-commencement framing directly into the
  date-state engine (so those two scripts emit the corrected form natively and the regen step becomes a
  pure cross-check) is the follow-up for v0.10.2.
- `event_calendar.json` / `api/events.json`: the only "GENIUS is already in force" strings that remain
  are the **correct CLARITY-basis note** (issuer layer in force; CLARITY is the pending market-structure
  layer). These are accurate and intentionally unchanged.

Per the register's standing discipline, this residual is declared rather than hidden.
